# Database rules

## Общие правила

- Не менять схему БД без отдельного подтверждения.
- Перед изменением моделей проверить связи, индексы, уникальные ограничения и миграции.
- Не удалять поля без плана миграции данных.
- Не использовать destructive migrations без подтверждения.
- Для production-данных предпочтительно использовать soft delete, если бизнес-логика это допускает.

## Prisma

- После изменения Prisma schema запускать prisma validate.
- После изменения Prisma schema запускать prisma generate.
- Проверять, что имена полей в коде соответствуют Prisma schema.
- Не смешивать raw SQL и Prisma без необходимости.

## Данные событий

- У событий должны быть устойчивые идентификаторы.
- Источник события должен сохраняться.
- Импортированные события должны защищаться от дублей.
- Вручную удалённые события не должны появляться снова без отдельной логики.
