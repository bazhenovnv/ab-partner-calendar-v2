# Backend rules

## Общие правила

- Перед изменением backend изучить связанные modules, controllers, services, DTO и Prisma schema.
- Не менять API-контракты без отдельного подтверждения.
- Не использовать raw SQL, если можно использовать ORM/Prisma.
- Если raw SQL необходим — объяснить причину и риски.
- Не создавать отдельные подключения к БД, если уже есть общий сервис подключения.
- Ошибка одной записи не должна останавливать весь импорт или синхронизацию.
- Внешние API должны иметь обработку ошибок и, где возможно, timeout.

## Синхронизация событий

- Импорт событий должен быть идемпотентным.
- Повторный запуск синхронизации не должен создавать дубли.
- Удалённые вручную события не должны повторно импортироваться без отдельной логики восстановления.
- Любая логика синхронизации должна сохранять источник события.

## Логирование

- Не логировать секреты, токены, cookies, env-переменные и персональные данные.
- Ошибки логировать так, чтобы было понятно место сбоя, но без утечки приватных данных.
